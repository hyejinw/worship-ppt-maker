# 가사 저장 전략 고민 정리

## 배경

현재 서비스의 가사 흐름은 대략 아래와 같다.

1. 사용자가 곡명을 검색한다.
2. DB에 저장된 가사가 있으면 그것을 반환한다.
3. DB에 없으면 Tavily + LLM으로 웹에서 가사를 수집해 반환한다.
4. 사용자는 Step1/Step2/Step3를 거치며 가사를 수정할 수 있다.
5. 실제 DB 저장은 현재 PPT 생성 시점에 이뤄진다.

문제는 여기서 사용자의 수정 빈도가 생각보다 높다는 점이다.

- 웹에서 가져온 가사는 오타, 누락, 줄바꿈 문제, 반복 구간 문제를 자주 가진다.
- 사용자는 예배 현장에 맞게 가사를 줄이거나, 반복을 정리하거나, 특정 버전으로 바꾸는 경우가 많다.
- 따라서 "검색 직후 가져온 가사"와 "사용자가 최종적으로 쓰기로 결정한 가사"는 의미가 다르다.

이 둘을 같은 데이터로 취급하면 DB 품질이 금방 흔들릴 수 있다.

## 핵심 질문

고민은 크게 두 가지다.

1. 검색 직후 최초 수집본을 DB에 넣을 것인가?
2. 사용자가 수정한 최종 가사를 DB에 넣을 것인가?

둘 다 장단점이 있다.

## 옵션별 분석

### 옵션 A. 검색 직후 최초 수집본만 저장한다

의미:

- 웹에서 가져온 결과를 "캐시"처럼 저장한다.
- 다음 검색부터는 같은 곡을 다시 Tavily/Groq로 호출하지 않는다.

장점:

- API 비용과 응답 시간을 줄일 수 있다.
- 같은 곡을 여러 번 검색할 때 일관된 결과를 준다.
- PPT 생성 전에도 DB가 조금씩 축적된다.

단점:

- 잘못 수집된 가사가 그대로 DB에 남을 수 있다.
- 수집본이 검증되지 않았는데도 "DB에 있으니 신뢰 가능한 데이터"처럼 보일 수 있다.

결론:

- 저장 자체는 유용하지만, "정답 데이터"가 아니라 "수집 캐시"로 취급해야 한다.

### 옵션 B. 사용자가 수정한 최종 가사를 기존 lyrics 테이블에 덮어쓴다

의미:

- 한 사용자가 손본 가사를 곧바로 공용 정답으로 승격한다.

장점:

- 수정된 더 좋은 가사가 축적될 가능성이 있다.
- 시간이 지날수록 데이터가 정제될 수도 있다.

단점:

- 모든 수정이 공용 정답은 아니다.
- 사용자는 예배 순서에 맞춰 일부 구절을 빼거나 반복을 조정할 수 있다.
- 개인 스타일, 교회 스타일, 임시 편집이 그대로 공용 데이터에 반영될 수 있다.
- 실수로 잘못 수정한 내용도 원본을 오염시킨다.

결론:

- 현재 단계에서는 위험하다.
- 사용자의 편집본을 곧바로 공용 lyrics 레코드에 덮어쓰는 방식은 권장하지 않는다.

### 옵션 C. 최초 수집본과 사용자 수정본을 분리 저장한다

의미:

- 최초 수집본은 원본/캐시로 저장한다.
- 사용자 수정본은 별도 저장한다.

장점:

- 원본과 수정본의 의미를 분리할 수 있다.
- 데이터 품질을 지키면서도 사용자의 수정 가치를 보존할 수 있다.
- 나중에 "개인 수정본", "검수된 공용 수정본", "가장 많이 사용된 수정본" 같은 전략으로 확장 가능하다.

단점:

- 스키마와 조회 우선순위가 조금 복잡해진다.
- 구현 비용이 옵션 A/B보다 높다.

결론:

- 가장 현실적이고 장기적으로 안전한 구조다.
- 현재 제품 방향에도 가장 잘 맞는다.

## 추천 방향

### 한 줄 요약

`수집본은 캐시, 수정본은 사용자 자산`으로 분리한다.

### 권장 정책

1. 최초 웹 수집본은 DB에 저장한다.
2. 다만 공용 정답이 아니라 "imported/raw" 성격으로 저장한다.
3. 사용자가 수정한 가사는 별도 저장한다.
4. 검색 시에는 사용자 수정본을 우선하고, 없으면 최초 수집본을 사용한다.
5. 수정본이 공용 정답이 되려면 별도 기준이나 검수 과정을 둔다.

## 추천 데이터 모델

### 1) 기존 lyrics 테이블은 원본/캐시 테이블로 유지

예상 필드:

- `id`
- `song_title`
- `title_normalized`
- `artist`
- `lyrics`
- `source` (`db`, `tavily`, `manual`)
- `status` (`imported`, `verified`, `curated`)
- `created_at`
- `updated_at`

의도:

- 검색 직후 저장되는 가사는 보통 `source=tavily`, `status=imported`
- 이후 검증되거나 관리자가 정제한 경우 `status=verified` 또는 `curated`

### 2) 사용자 수정본 전용 테이블 추가

예시 테이블명:

- `lyrics_edits`

예상 필드:

- `id`
- `song_title`
- `title_normalized`
- `artist`
- `base_lyrics_id` (어떤 원본에서 시작했는지)
- `edited_lyrics`
- `session_id` 또는 `user_id`
- `is_final_used` (실제로 PPT 생성에 사용된 편집본인지)
- `created_at`
- `updated_at`

의도:

- 사용자가 중간에 손본 편집본을 저장 가능
- 실제 생성에 사용된 최종 가사도 저장 가능
- 원본 오염 없이 개인 편집 히스토리를 남길 수 있음

## 조회 우선순위 제안

같은 곡을 다시 검색할 때 추천 우선순위:

1. 현재 사용자(session/user)의 최신 수정본
2. 검증된 공용 가사 (`verified` 또는 `curated`)
3. 최초 수집본 (`imported`)
4. 아무것도 없으면 Tavily + LLM 재검색

이 우선순위를 쓰면:

- 내 수정본을 계속 재사용할 수 있고
- 공용 데이터 품질도 유지되며
- 외부 API 비용도 절약된다.

## 저장 시점 제안

### 추천안

#### A. 검색 직후

- 웹 수집 성공 시 최초 수집본을 저장
- 단, status는 `imported`

#### B. 사용자가 실제 PPT 생성 버튼을 누를 때

- 그 시점의 최종 가사를 `lyrics_edits`에 저장
- `is_final_used=true`

이 구조가 좋은 이유:

- "검색해봤지만 안 쓴 가사"와
- "실제로 내가 검토하고 최종 사용한 가사"를 구분할 수 있다.

## 왜 수정본을 바로 공용 테이블에 넣으면 안 되는가

### 실제로 발생 가능한 케이스

- 사용자가 후렴 반복을 예배 순서에 맞게 삭제함
- 브릿지 한 구간을 생략함
- 자막용 줄바꿈만 손봄
- 특정 아티스트/버전의 가사로 바꿈
- 임시 오타를 낸 채 생성함

이 편집본을 그대로 공용 정답으로 저장하면, 다음 사용자에게는 오히려 잘못된 데이터가 된다.

즉 수정본은 가치가 높지만, 곧바로 공용 정답이라고 가정하면 안 된다.

## 단계별 도입안

### 1단계: 가장 현실적인 최소 변경

- 검색 직후 최초 수집본 저장
- 사용자가 최종 생성한 가사는 `lyrics_edits`에만 저장
- 검색 시 session 기준 수정본 우선 조회

장점:

- 구현 난이도가 비교적 낮다.
- 지금 가장 필요한 "내가 고친 가사를 다시 쓰는 문제"를 해결한다.

### 2단계: 수정본 재사용 고도화

- 같은 곡에 대한 수정본이 여러 개 쌓일 수 있게 함
- 최신 사용본 우선 제공
- 사용자가 "원본 불러오기 / 내 수정본 불러오기" 선택 가능

### 3단계: 공용 정답 승격

- 반복 사용 횟수
- 관리자의 승인
- 다수 세션의 일치도

같은 기준으로 일부 수정본만 `verified` 또는 `curated` 공용 데이터로 승격

## 현재 코드 기준으로 봤을 때의 제안

현재는 검색 시점에 저장하지 않고, PPT 생성 시점에만 DB 저장을 하고 있다.

이 구조는 아래 한계를 가진다.

- 검색만 하고 나간 곡은 DB에 안 쌓인다.
- 이미 한 번 잘 찾은 곡도 다시 외부 API를 탈 수 있다.
- 수정본과 원본의 구분 없이 한 번에 저장 흐름이 섞이기 쉽다.

따라서 추천 리팩터링 방향은:

1. `/api/lyrics/search` 성공 시 imported 가사 저장
2. `/api/ppt/generate` 시점에는 최종 사용 가사를 edits 테이블에 저장
3. 이후 검색 API는 edits 우선 조회 로직 추가

## 최종 결론

현재 상황에서 가장 적절한 정책은 아래다.

- 최초 불러온 가사는 저장한다.
- 그러나 그것은 "수집 원본/캐시"로 저장한다.
- 사용자가 수정한 가사는 별도 저장한다.
- 수정본으로 공용 lyrics 레코드를 덮어쓰지 않는다.
- 검색 시에는 개인 수정본을 우선 사용한다.

즉,

`원본과 편집본을 분리 저장하는 구조가 지금 가장 안전하고, 앞으로도 확장성이 좋다.`
